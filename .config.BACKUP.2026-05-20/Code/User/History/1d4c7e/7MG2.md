# Design: IA Chat — Visión de Producción Final

> **Propósito**: Este documento describe la visión completa de IA Chat como una aplicación de escritorio terminada, lista para producción. No es un MVP ni una progresión por fases — es el destino final: un orquestador conversacional multiagente maduro, seguro, extensible y preparado para el uso diario.
>
> **Stack**: Rust · Tauri · Leptos · AI SDK.rs · llama.cpp

---

## Arquitectura

### Visión General

IA Chat es una **aplicación de escritorio nativa** (Tauri) con una **arquitectura hexagonal**, donde el núcleo de dominio (agentes, herramientas, contexto) está aislado de los adaptadores de infraestructura (LLMs, sistema de archivos, red, UI).

### Capas

| Capa                | Responsabilidad                                       | Tecnología                        |
| ------------------- | ----------------------------------------------------- | --------------------------------- |
| **UI Layer**        | Renderizado, interacción, animaciones                 | Leptos (SSR/WASM) + Tauri WebView |
| **Store Layer**     | Estado global, sincronización reactiva                | Leptos signals + Tauri state      |
| **Orquestación**    | Máquina de estados del chat, enrutamiento de mensajes | Rust nativo                       |
| **Agentes**         | Definición, registro, ejecución, delegación           | AI SDK.rs + FSM                   |
| **Herramientas**    | Ejecución de acciones (archivos, terminal, red, MCP)  | Rust nativo + trait system        |
| **Providers**       | Abstracción de LLMs locales y cloud                   | AI SDK.rs + Provider trait        |
| **Infraestructura** | Memoria, logging, gestión de modelos, seguridad       | Rust nativo                       |

### Principios Arquitectónicos

- **Hexagonal**: El dominio (agentes, herramientas) no conoce a los adaptadores (UI, LLMs, filesystem)
- **Immutable State**: Cada transición de estado produce un nuevo estado, no muta el anterior
- **Fail Fast**: Errores de tool, provider o red se propagan con estado ERROR explícito
- **Secure by Default**: Terminal sandboxeado, write tool con confirmación, MCP con permisos
- **Pluggable**: Tools, providers, skills y agentes se registran dinámicamente sin tocar el núcleo

```
IDLE ──► LISTENING ──► THINKING ──► WORKING ──► STREAMING ──► IDLE
  ▲                                                        │
  │                        ┌───────────────────────────────┘
  │                        ▼
  └────────────────────── ERROR
```

| Estado       | Visual                                 | Descripción                            |
| ------------ | -------------------------------------- | -------------------------------------- |
| `IDLE`       | Círculo verde estático                 | Chat listo, sin actividad              |
| `LISTENING`  | Onda de sonido animada                 | Usuario escribiendo o mensaje recibido |
| `THINKING`   | Círculo pulsante amarillo              | Main Agent analizando la solicitud     |
| `WORKING`    | Spinner rotando azul                   | Ejecutando herramientas                |
| `DELEGATING` | Árbol expandiéndose                    | Main Agent delegando a sub-agentes     |
| `STREAMING`  | Cursor parpadeante + texto apareciendo | Recibiendo respuesta del LLM           |
| `ERROR`      | Círculo rojo con X                     | Error en cualquier etapa               |

#### Sistema de Temas

- **Tema claro/oscuro** con detección automática del sistema
- **Tokens de diseño** definidos como variables CSS (spacing, radii, colors, typography)
- **Modo compacto** opcional para usuarios que prefieren más densidad de información

### Diseño de Datos

```rust
// ─── Mensajes ───────────────────────────────────────────────
struct Message {
    id: Uuid,
    role: MessageRole,        // User | Assistant | Tool | System
    content: MessageContent,  // Text | ToolCall | ToolResult | Error
    timestamp: DateTime<Utc>,
    status: MessageStatus,    // Pending | Streaming | Completed | Failed
    parent_id: Option<Uuid>,  // Para hilos y delegación
    metadata: MessageMetadata, // tokens, modelo, duración
}

// ─── Estados ────────────────────────────────────────────────
struct ChatState {
    status: ChatStatus,
    messages: Vec<Message>,
    current_model: String,
    context_tokens: usize,
    active_tools: Vec<String>,
    active_subagents: Vec<AgentInvocation>,
    error: Option<ChatError>,
}

// ─── Agentes ────────────────────────────────────────────────
struct AgentConfig {
    name: String,
    description: String,
    model: String,
    max_steps: u32,
    system_prompt: String,
    tools: Vec<String>,
    subagents: Vec<String>,
    guardrails: Option<Guardrails>,
}

// ─── Providers ──────────────────────────────────────────────
trait LLMProvider: Send + Sync {
    fn name(&self) -> &str;
    fn capabilities(&self) -> ProviderCapabilities;
    async fn connect(&mut self) -> Result<(), ProviderError>;
    async fn stream_chat(&self, req: ChatRequest) -> Result<ChatStream, ProviderError>;
    async fn list_models(&self) -> Result<Vec<ModelInfo>, ProviderError>;
    async fn pull_model(&self, id: &str) -> Result<ProgressStream, ProviderError>;
}

// ─── Tools ──────────────────────────────────────────────────
trait Tool: Send + Sync {
    fn name(&self) -> &str;
    fn description(&self) -> &str;
    fn schema(&self) -> ToolSchema;
    async fn execute(&self, params: Value) -> Result<ToolResult, ToolError>;
}
```

## Features

### 1. Chat Conversacional Full

- Mensajes de usuario y asistente con **streaming token a token**
- **Markdown GFM** completo: headings, bold, italic, listas, tablas, code blocks con syntax highlighting
- **Tool calls visibles**: cada invocación de herramienta se muestra como una tarjeta expandible
- **Edición de mensajes**: editar un mensaje previo regenera la conversación desde ese punto
- **Cancelación**: botón para detener la generación en curso
- **Historial persistente**: las conversaciones se guardan y restauran entre sesiones

### 2. Sistema Multiagente (Main + Sub-agentes)

- **Main Agent**: único punto de contacto con el usuario. Analiza, responde directamente o delega
- **Sub-agentes** definidos en `.mi_suit/agents/*.md`:
  - **Architect**: Diseño arquitectónico, especificaciones, planificación
  - **Coder**: Implementación, escritura de código, refactoring
  - **Reviewer**: Code review, validación, testing, calidad
- Delegación visual: el usuario ve el flujo completo (Main → Architect → Coder → Reviewer)
- **FSM con guards**: cada sub-agente tiene una máquina de estados con pre/post-validaciones
- Retorno de control: el sub-agente reporta resultado y el Main Agent continúa

### 3. Sistema de Skills

- Skills reutilizables en `.mi_suit/skills/*/SKILL.md`
- Cargables por agentes en runtime
- Catálogo navegable desde la UI
- Compatibilidad con MCP para skills externos

### 4. Herramientas del Sistema

| Herramienta       | Descripción                         | Seguridad                                |
| ----------------- | ----------------------------------- | ---------------------------------------- |
| **Read Tool**     | Leer archivos del proyecto          | Solo lectura, no modifica                |
| **Write Tool**    | Escribir/editar archivos            | Confirmación requerida, diff preview     |
| **Grep Tool**     | Búsqueda de contenido en archivos   | Solo lectura                             |
| **Glob Tool**     | Búsqueda de archivos por patrón     | Solo lectura                             |
| **Web Search**    | Búsqueda en internet                | Sandbox, timeout configurable            |
| **Terminal Tool** | Ejecución de comandos               | Sandbox con whitelist/blacklist, timeout |
| **Git Tool**      | Operaciones git (status, diff, log) | Solo consulta por defecto                |
| **MCP Client**    | Conexión a servidores MCP externos  | Permisos por server                      |

### 5. Proveedores de LLM

- **Local**: llama.cpp (GGUF nativo, sin proceso externo) + Ollama (compatibilidad)
- **Cloud**: OpenAI, Anthropic, Google, OpenRouter, xAI, 70+ providers via AI SDK.rs
- **Model Manager**: descarga, cacheo y gestión de modelos GGUF desde HuggingFace
- **Catálogo de modelos**: navegable desde la UI, con recomendaciones por caso de uso
- **API REST opcional**: el provider local puede exponer API compatible con Ollama

### 6. Provider Dashboard

- **Métricas en vivo**: tokens/segundo, latencia de primer token, costo estimado
- **Histórico de consumo**: charts de uso por día/semana/mes
- **Comparativa de modelos**: velocidad vs. calidad vs. costo
- **Alertas**: umbrales de costo configurables

### 7. Context Manager Inteligente

- **Buffer circular** con evictón LRU, max tokens configurable
- **Preservación contextual**: mantiene mensajes recientes + aquellos marcados como importantes
- **Resumen automático**: cuando el contexto se llena, resume mensajes antiguos en lugar de descartarlos
- **Memoria persistente**: integración con engram/memoria atómica para retención entre sesiones

### 8. Seguridad y Gobernanza

- **Terminal sandboxeado**: whitelist de comandos permitidos, blacklist de comandos peligrosos, timeout forzoso
- **Write confirmado**: toda escritura de archivos muestra diff y requiere confirmación
- **MCP con permisos**: cada server MCP tiene alcance definido (leer, escribir, ejecutar)
- **Audit log**: todas las herramientas invocadas quedan registradas con timestamp y resultado
- **Límites de recursos**: max steps por agente, max tokens por request, timeout global

### 9. Input Enhancer

- Corrección ortográfica antes de enviar al LLM
- Sugerencias de autocompletado basadas en contexto
- Detección de comandos vs. conversación natural

### 10. Experiencia de Desarrollo (DX)

- **Hot reload** de agentes y skills (sin recompilar)
- **Modo debug**: panel que muestra tokens, estado interno, llamadas a herramientas
- **Exportación**: exportar conversación a markdown, JSON, o PDF
- **Atajos de teclado**: navegación completa sin mouse

## Producción

### Rendimiento

| Métrica                         | Target               |
| ------------------------------- | -------------------- |
| Startup time                    | < 2s                 |
| Primer token (local)            | < 1s (cuantizado)    |
| UI responsivo durante streaming | Sin bloqueo          |
| Memoria (chat activo)           | < 150MB              |
| Memoria (ocioso)                | < 60MB               |
| Tamaño binario                  | < 30MB (sin modelos) |

### Seguridad

- **Sandbox de terminal**: lista blanca configurable, prohibidos comandos de red/privilegio
- **Write con diff**: el usuario ve el diff antes de confirmar cualquier escritura
- **MCP permissions**: ámbito definido por server, no por tool individual
- **Tauri security model**: isolation pattern, capability-based permissions
- **No ejecución remota**: todos los LLMs locales, los cloud requieren API key del usuario

### Extensibilidad

- Nuevos **tools**: implementar `trait Tool`, registrar en `ToolRegistry`
- Nuevos **providers**: implementar `trait LLMProvider`, registrar en `ProviderRegistry`
- Nuevos **agentes**: crear archivo `.mi_suit/agents/<name>.md`
- Nuevos **skills**: crear archivo `.mi_suit/skills/<name>/SKILL.md`
- Nuevos **MCP servers**: conectar via protocolo estándar MCP
